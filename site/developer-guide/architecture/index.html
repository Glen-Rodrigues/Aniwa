    Architecture¶

    This document explains the architecture of Aniwa, including:

    • system structure
    • execution flow
    • core modules
    • profiling pipeline
    • reporting systems
    • scalability strategy
    • future architectural direction

    Aniwa is designed as:

    modular universal data profiling infrastructure

    not merely:

    a CLI utility

    Purpose of the Architecture¶

    The architecture exists to ensure:

    • scalability
    • maintainability
    • extensibility
    • performance
    • contributor friendliness

    Architectural Philosophy¶

    Aniwa architecture is built around several principles:

    Principle
    modular
    scalable
    developer-first
    extensible
    automation-friendly
    maintainable

    Core Philosophy¶

    Aniwa is designed around:

    separation of concerns

    This means each system should own:

    one primary responsibility

    High-Level Architecture¶

    Aniwa currently consists of several major layers:

    CLI
→ Configuration
→ Readers
→ Profiling Engine
→ Models
→ Reports
→ Templates

    High-Level Execution Flow¶

    The current execution pipeline:

    User Command
→ CLI Parsing
→ Config Resolution
→ Dataset Reading
→ Profiling Engine
→ Insight Generation
→ Report Rendering
→ Output Export

    The configuration system handles:
